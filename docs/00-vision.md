# 00 — Vision

## Цель
Backend-сервис для iOS-приложения, который оркеструет диалог пользователя с Claude (Anthropic API), управляет политикой доступа, учитывает кредиты (внутренняя валюта), безопасно хранит пользовательские API-ключи (BYOK), реализует tool-loop с клиентом iOS и ведёт аудит.

## Ключевые ценности (приоритеты архитектуры)
1. **Корректность бизнес-правил** доступа и биллинга (атомарность, идемпотентность).
2. **Безопасность** секретов (BYOK at-rest encryption, отсутствие утечек в логи).
3. **Простота** — монолит на FastAPI, без преждевременной микросервисной декомпозиции.
4. **Наблюдаемость** — метрики/логи/трейсы с первого дня.

## Бизнес-правила (обязательные, источник истины для Policy Engine)
| # | Правило |
|---|---|
| BR-1 | Без активной подписки доступен ровно **1 trial message за всё время жизни** аккаунта. |
| BR-2 | С активной подпиской генерация разрешена; при `mode=credits` происходит списание кредитов. |
| BR-3 | Если `creditsBalance = 0` → генерация при `mode=credits` заблокирована (`credits_empty`). |
| BR-4 | `mode=byok` разрешён **только при активной подписке** и включённом валидном BYOK. |
| BR-5 | Если подписка неактивна/истекла → генерация запрещена **всегда**, включая BYOK (`subscription_expired` / `subscription_required`). |
| BR-6 | `/v1/policy/effective` всегда консистентен с фактическим решением `/v1/chat/run` (один и тот же Policy Engine). |
| BR-7 | Все мутирующие tool-действия и все списания кредитов фиксируются в audit. |

### Приоритет blockReason (порядок проверок Policy Engine)
0. `rate_limited` — **gateway-concern**: проверяется на API Gateway **до** Policy Engine и всегда возвращается как HTTP `429`. **НЕ** часть `evaluate()` (ADR-002), **НЕ** приходит как `status=blocked` body и **НЕ** входит в `/policy/effective.reasons[]` (BLK-7b, [policy-engine/02-api-contracts.md](modules/policy-engine/02-api-contracts.md)). Приведён здесь только как самый ранний барьер транспортного слоя.
1. `subscription_required` / `subscription_expired` (BR-5)
2. Для `mode=byok`: `byok_disabled` → `byok_invalid` (BR-4)
3. Для `mode=credits`: `trial_used` (если нет подписки и trial израсходован, BR-1) → `credits_empty` (BR-3)
4. `policy_denied` (общий fallback)

> Проверки 1–4 — это собственно перечень policy-проверок (`evaluate`). Пункт 0 вынесен в gateway-слой и не относится к Policy Engine.

## Целевая аудитория
Клиент — единое iOS-приложение (один тип клиента). Multi-tenant в смысле «много пользователей», но один продукт.

## NFR (нефункциональные требования)
| NFR | Цель |
|---|---|
| Латентность | `chat_run_latency_p95` ≤ 1.5s сверх времени ответа Anthropic (только оркестрация/политика/БД). |
| Доступность | Single-region, целевой uptime 99.5% на старте. |
| Согласованность биллинга | Строгая: списание атомарно и идемпотентно, отрицательный баланс невозможен. |
| Безопасность секретов | BYOK ключи зашифрованы at-rest (envelope encryption), не попадают в логи/трейсы/ответы. |
| Масштаб на старте | До ~10k активных пользователей, вертикальное масштабирование + 2–3 реплики API. |
| Idempotency | `/wallet/consume` и `/chat/tool-result` идемпотентны по ключу. |

## Out of scope (на старте)
- Web/Android клиенты.
- Биллинг через Stripe/web (только StoreKit / Apple subscriptions).
- Серверное исполнение iOS tools (files/calendar/reminders) — выполняет клиент.
- Multi-region / active-active.

## Acceptance Criteria (backend)
- AC-1: Trial доступен ровно 1 раз lifetime без подписки.
- AC-2: При неактивной подписке `/chat/run` всегда `blocked` (включая BYOK).
- AC-3: Списание кредитов атомарно и идемпотентно; отрицательный баланс невозможен.
- AC-4: Tool-loop стабильно поддерживает несколько шагов подряд в рамках сессии.
- AC-5: BYOK хранится зашифрованно и не утекает в логи.
- AC-6: `/policy/effective` всегда консистентен с решением `/chat/run`.
- AC-7: Аудит фиксирует все мутирующие tool-действия и все списания.
