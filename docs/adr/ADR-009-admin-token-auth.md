# ADR-009 — Admin-авторизация: изолированный admin-токен (статический secret через заголовок)

- Статус: Accepted
- Дата: 2026-06-01
- Связан с: [ADR-001](ADR-001-stack-choice.md), [ADR-007](ADR-007-lazy-user-provisioning.md), [05-security.md](../05-security.md), [modules/admin/](../modules/admin/README.md)

## Context

Появляется первый admin-сценарий: операторская/саппорт-функция начисления кредитов пользователю
(`POST /v1/admin/wallet/grant`), переиспользующая существующий идемпотентный `WalletService.grant()`
(`src/app/wallet/service.py:174`).

До этого ([05-security.md](../05-security.md) §Авторизация) система имела **единственную роль** `user`:
каждый запрос ограничен ресурсами `sub` из пользовательского JWT (RS256, доверенный issuer, [ADR-007](ADR-007-lazy-user-provisioning.md)).
Admin-действие принципиально иное: оператор действует **над чужим** `userId`, что прямо запрещено
пользовательской моделью (сверка `userId` тела с `sub` → `403`, [05-security.md](../05-security.md)).

Нужна авторизация admin-API, которая:
- **изолирована** от пользовательской auth (отдельный секрет/issuer, отдельная зависимость);
- **не допускает эскалации**: компрометация пользовательского JWT не даёт admin-прав, и наоборот;
- **не запускает пользовательский provisioning** ([ADR-007](ADR-007-lazy-user-provisioning.md)) и **не трогает trial**;
- учитывает **ротацию** секрета.

Рассмотрены два варианта (см. Alternatives): (A) статический `ADMIN_API_SECRET` через заголовок,
(B) отдельный admin-JWT issuer + ключ.

## Decision

**Статический `ADMIN_API_SECRET`, передаваемый в выделенном заголовке `X-Admin-Token`.**

1. **Секрет.** Высокоэнтропийный (≥ 32 байта, генерируется CSPRNG) `ADMIN_API_SECRET` хранится в
   secret manager / env, **никогда** в коде/репозитории/образе ([05-security.md](../05-security.md) §Секреты).
   Не пересекается с `JWT`-ключами, `KMS`, `ANTHROPIC_API_KEY`, `PREVIEW_URL_SECRET`.

2. **Транспорт.** Заголовок `X-Admin-Token: <secret>`. Отдельный от пользовательского `Authorization: Bearer <JWT>`.
   Admin-эндпоинты **не принимают** пользовательский JWT для авторизации действия; пользовательский `Authorization`
   на admin-роутах игнорируется как фактор авторизации.

3. **Зависимость `require_admin`.** Новая FastAPI-зависимость, **полностью отдельная** от `get_current_user`:
   - сравнивает `X-Admin-Token` с `ADMIN_API_SECRET` **constant-time** (`hmac.compare_digest`);
   - отсутствие/несовпадение → `401` (без раскрытия причины);
   - **НЕ выполняет** lazy-provisioning ([ADR-007](ADR-007-lazy-user-provisioning.md)), **НЕ читает/не трогает** `users.trial_used`,
     **НЕ создаёт** строку `users` для actor'а (admin — не пользователь системы);
   - не имеет `sub`/`userId`: actor admin-запроса фиксируется как `actor="admin"` в audit, без персональной идентичности.

4. **Невозможность эскалации.** Пользовательский путь (`get_current_user`) и admin-путь (`require_admin`)
   не пересекаются: разные секреты, разные заголовки, разные зависимости. Валидный пользовательский JWT
   не даёт доступа к `/v1/admin/*`; валидный admin-токен не даёт доступа к пользовательским ресурсам через
   пользовательские эндпоинты (там по-прежнему требуется JWT и сверка `sub`).

5. **Ротация.** Поддержка **двух** активных секретов на время ротациии: `ADMIN_API_SECRET` (основной) и
   опциональный `ADMIN_API_SECRET_PREV` (предыдущий, на grace-период). `require_admin` принимает совпадение
   с любым из непустых; оба сравниваются constant-time. После ротации `ADMIN_API_SECRET_PREV` очищается.
   Это позволяет менять секрет без окна отказа admin-API.

6. **Защита admin-API.** Отдельный, более строгий rate limit (per source IP, дефолт 10 req/min — конфигурируемо),
   strict Pydantic-валидация (`extra='forbid'`), size-лимиты тела (≤ 8 KB для admin-grant). Сам секрет
   **никогда не логируется** (redaction уже покрывает `*token*`/`*secret*`, заголовок `X-Admin-Token` добавляется
   в redaction allowlist явно).

## Consequences

**Положительные:**
- Простейшая работоспособная модель: один секрет, один заголовок, одна зависимость. Соответствует §00-vision (простота).
- Жёсткая изоляция от пользовательской auth: нет общих ключей, нет общего кода, эскалация невозможна by construction.
- Не запускается provisioning/trial — admin не загрязняет пользовательские инварианты.
- Ротация без downtime (две версии секрета).

**Отрицательные / ограничения:**
- Нет персональной идентичности оператора (нет «кто именно из админов»): actor — обезличенный `admin`.
  Приемлемо для текущего этапа (узкий круг операторов, действие аудируется с `reason`). При появлении нескольких
  операторов с требованием атрибуции — переход на вариант (B) отдельным ADR (см. [Q-009-1](../99-open-questions.md)).
- Статический секрет требует операционной дисциплины ротации (нет авто-`exp`, как у JWT).
- Нет встроенного scope/least-privilege на уровне токена (один секрет = все admin-операции). Допустимо при единственной
  admin-операции (grant); при росте admin-surface — пересмотр.

## Alternatives

1. **Отдельный admin-JWT issuer + ключ (вариант B).** Отдельная пара ключей/issuer, admin-JWT с `exp` и, возможно,
   claim актёра. Даёт авто-истечение и атрибуцию оператора. Отвергнуто на этом этапе: избыточная инфраструктура
   (отдельный issuer/JWKS, выпуск admin-токенов) при единственной admin-операции и узком круге операторов.
   Сохранено как путь миграции при росте требований ([Q-009-1](../99-open-questions.md)).
2. **Роль `admin` в пользовательском JWT (claim `role`/`scope`).** Прямо отвергнуто решением пользователя и из
   соображений изоляции: смешивает пользовательскую и admin-идентичности, компрометация пользовательского issuer →
   эскалация до admin, нарушает «identity из JWT = только владелец своих ресурсов» ([ADR-007](ADR-007-lazy-user-provisioning.md)).
3. **mTLS / network-level isolation admin-API.** Сильнее, но операционно тяжелее (управление клиентскими сертификатами)
   и не отменяет необходимость app-level проверки. Может быть добавлен поверх как defense-in-depth позже; не требуется на старте.
