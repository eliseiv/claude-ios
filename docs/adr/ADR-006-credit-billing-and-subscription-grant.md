# ADR-006 — Биллинг кредитов: 1 кредит = 1 сообщение; фиксированный пакет при подписке

- Статус: Accepted
- Дата: 2026-05-21
- Связанные: [ADR-002](ADR-002-access-policy-state-machine.md), [ADR-005](ADR-005-idempotency-ledger.md)
- Закрывает: [Q-004-1](../99-open-questions.md), [Q-006-1](../99-open-questions.md)

## Context

Две связанные открытые продуктовые неопределённости блокировали финализацию биллинга:

- **Q-004-1** — формула конвертации usage (input/output tokens) → кредиты для `mode=credits`.
  Без согласованной семантики нельзя реализовать списание (`consume`).
- **Q-006-1** — механика начисления кредитов при подписке (grant): фиксированный пакет
  на период vs иные модели.

Token-based тарификация (списание пропорционально usage по коэффициентам на модель)
рассматривалась, но усложняет UX (непредсказуемая стоимость сообщения для пользователя),
требует поддержки тарифной таблицы на модель и её версионирования, и усложняет тестируемость
списания. Для текущего этапа продукта это избыточная сложность (см. §00-vision — простота).

## Decision

### 1. Кредитная стоимость генерации: 1 кредит = 1 сообщение (Q-004-1)

- При `mode=credits` один **завершённый пользовательский message-шаг** списывает **ровно 1 кредит**
  через `POST /v1/wallet/consume` с `amount=1`.
- «Завершённый message-шаг» = успешная генерация **assistant_message** (ответ с `end_turn`/текстом).
- Промежуточные `tool_call`-раунды (когда модель возвращает `tool_use` и ожидает `tool-result`)
  **НЕ списывают** кредиты. Tool-loop из нескольких раундов в рамках одного пользовательского
  сообщения завершается **одним** списанием на финальном assistant_message.
- Идемпотентность списания привязана к `messageStepId` — отдельному идентификатору пользовательского
  message-шага, который Orchestrator генерирует в `/chat/run` при старте нового шага и переиспользует
  для всех его tool-раундов (включая re-entry из `/chat/tool-result`) вплоть до финального assistant_message.
  `messageStepId` персистируется в `chat_steps.message_step_id` и `tool_calls.message_step_id`.
  В публичный контракт `/wallet/consume` Orchestrator передаёт `messageStepId` в поле `requestId (idempotency key)`
  (ТЗ §4.5); повторное списание по тому же `messageStepId` не выполняется (ADR-005).
  Это гарантирует «1 списание на 1 пользовательский message-шаг» независимо от числа tool-раундов.
  **Важно:** `messageStepId` — это НЕ gateway correlation `requestId` (`X-Request-Id`, per-HTTP-request, логи/трейсы);
  последний для биллинга не используется (ADR-005, раздел «Терминология»).
- `usage` (inputTokens/outputTokens/model, cacheReadTokens/cacheWriteTokens) сохраняется в
  `ledger_transactions.meta` и `chat_steps.usage` исключительно для аудита и аналитики.
  На величину `amount` он **не влияет** (`amount` всегда `1`).

### 2. Начисление кредитов при подписке: фиксированный пакет на период (Q-006-1)

- Активация или продление подписки (через `POST /v1/subscription/sync` с переходом в `active`
  на новый период) начисляет **фиксированное** число кредитов как `credit`-транзакцию в ledger
  (`Wallet.grant`).
- Число кредитов на период — **конфигурируемое**, параметр `SUBSCRIPTION_CREDITS_PER_PERIOD`
  (env / pydantic-settings), дефолт **1000**. Магических чисел в коде нет.
- Начисление **идемпотентно по периоду**: ключ идемпотентности grant = `transactionId` периода
  (App Store transaction). Повторный `sync` той же транзакции/периода **не начисляет дважды**
  (ADR-005, unique index `ux_ledger_idempotency`).
- Кредиты не сгорают между периодами автоматически на текущем этапе (накопление баланса);
  политика истечения кредитов — потенциальный будущий tech-debt, не входит в это решение.

## Consequences

- (+) Предсказуемая стоимость для пользователя: 1 ответ = 1 кредит.
- (+) Простая, легко тестируемая семантика списания и начисления.
- (+) `amount` фиксирован → нет тарифных таблиц на модель, нет их версионирования.
- (+) usage сохраняется для аудита/аналитики без влияния на биллинг — данные не теряются.
- (−) Стоимость не отражает реальное потребление токенов (длинный ответ = тот же 1 кредит).
  Приемлемо для текущего этапа; при необходимости пересматривается отдельным ADR.
- (−) `SUBSCRIPTION_CREDITS_PER_PERIOD` требует согласованного значения в prod-конфиге.

## Alternatives

- **Token-based тарификация** (usage × коэффициент на модель). Отклонено: сложность UX, тарифные
  таблицы, версионирование, хуже тестируемость. Может быть введено позже отдельным ADR при
  появлении продуктовой потребности.
- **Начисление кредитов пропорционально цене плана** или разовый lifetime-grant. Отклонено в пользу
  фиксированного конфигурируемого пакета на период — проще и прозрачнее.
