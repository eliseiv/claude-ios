# ADR-032 — Контрактный дефолт `notificationsEnabled` → `false` (privacy-by-default)

- **Статус:** Accepted
- **Дата:** 2026-06-16
- **Связано:** [ADR-012](ADR-012-assistant-mode-vs-billing-mode.md) (таблица `user_preferences`, разведение `assistant_mode`/`billing_mode`), модуль [preferences](../modules/preferences/README.md), модуль [notifications](../modules/notifications/README.md), [TD-011](../100-known-tech-debt.md) (отправка push не реализована).

> Примечание о нумерации: номер **ADR-032** ранее не существовал (предыдущие принятые — [ADR-030](ADR-030-toolcallid-in-server-tools.md), [ADR-031](ADR-031-absolute-preview-url.md); 032 свободен). Берётся первый свободный номер.

## Context

Репорт iOS-разработчика: дефолт `notificationsEnabled` должен быть `false`. Правильный UX-flow на iOS — приложение сначала запрашивает системное разрешение на push, и только после его выдачи включает уведомления. При дефолте `true` сервер сообщает «уведомления включены» ещё до того, как пользователь дал системное разрешение, что создаёт рассинхронизацию: настройка «on», но push физически невозможны.

Текущее состояние (подтверждено по коду):
- `src/app/preferences/service.py` `_defaults()` возвращает `notifications_enabled=True` — **главная точка контракта**: именно это отдаёт `GET /v1/preferences` при отсутствии строки `user_preferences`.
- `src/app/models/tables.py` колонка `notifications_enabled` имеет `server_default=sa_text("true")`.
- Миграция `0004` создала колонку с `server_default="true"`; последняя ревизия — `0008_adapty_webhook_events`.
- Pydantic-схема `src/app/schemas/preferences.py` дефолтов в `Field` не задаёт (источник дефолта — сервис), менять её не нужно.
- Фактическая отправка push ещё не реализована ([TD-011](../100-known-tech-debt.md)) — на старте только хранение настройки и регистрация device-токена.

## Decision

Сменить **контрактный дефолт** `notificationsEnabled` с `true` на `false`. Меняется только дефолт для **новых / без-строки** пользователей (lazy-default сервиса + `server_default` колонки). Это согласуется с privacy-by-default и flow «системное разрешение iOS сначала, потом включение».

Явные ограничения решения:
1. **Существующие строки `user_preferences` НЕ трогаем.** Явный выбор пользователя (`notifications_enabled=true/false` в уже созданной строке) сохраняется. Миграция меняет **только** `server_default` колонки, **без** `UPDATE` существующих строк (без backfill).
2. **Lazy-default сервиса** (`_defaults()`) меняется на `False` — это то, что `GET` отдаёт при отсутствии строки. Это и есть главная точка наблюдаемого контракта.
3. **Схема не меняется** — дефолт приходит из сервиса.
4. **Семантика остального не меняется:** единый источник настройки — `user_preferences.notifications_enabled`; отдельного toggle-эндпоинта в notifications нет; перед будущей отправкой push (TD-011) уважается значение (выключено → не слать).

## Consequences

- Новый пользователь (или пользователь без строки `user_preferences`) получает `notificationsEnabled=false` из `GET /v1/preferences`. Клиент включает уведомления через `PATCH /v1/preferences` после получения системного разрешения.
- Усиливает private-by-default: до явного согласия пользователя push считаются выключенными. Согласуется с [TD-011](../100-known-tech-debt.md) (доставка ещё не реализована — безопасно стартовать с «выключено»).
- Существующие пользователи не затронуты: их сохранённый выбор не перезаписывается. На проде риск регрессии для текущих пользователей отсутствует.
- Без backfill миграция тривиальна и обратима (downgrade возвращает `server_default="true"`).

## Alternatives

- **Backfill существующих строк в `false`.** Отклонено: перезаписывает явный выбор пользователей, нарушает доверие к настройке, не требуется для цели (нужен только правильный дефолт для новых).
- **Менять только `server_default`, оставив сервисный lazy-default `true`.** Отклонено: lazy-default — главная точка наблюдаемого контракта (`GET` без строки); рассинхрон между БД-дефолтом и тем, что отдаёт API, недопустим.
- **Менять только сервисный lazy-default, оставив `server_default="true"`.** Отклонено: новая строка, созданная без явного `notifications_enabled` (теоретический путь вставки), получила бы `true` — расхождение БД ↔ контракт. Держим оба дефолта согласованными.
- **Оставить дефолт `true`.** Отклонено: ломает iOS-flow «разрешение сначала», даёт ложное «включено» без системного permission.
