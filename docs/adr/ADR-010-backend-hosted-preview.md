# ADR-010 — Backend-hosted preview сгенерированных сайтов (signed URL + threat model)

- Статус: Accepted
- Дата: 2026-06-01
- Связан с: [ADR-001](ADR-001-stack-choice.md), [ADR-009](ADR-009-admin-token-auth.md), [ADR-011](ADR-011-server-side-tools.md), [05-security.md](../05-security.md), [03-data-model.md](../03-data-model.md), [modules/website-builder/](../modules/website-builder/README.md)

## Context

Новая фича website-builder: Claude генерирует статический сайт (HTML/CSS/JS) по запросу пользователя.
Сгенерированные файлы нужно **хранить** и **показывать в превью** так, чтобы превью работало в любом браузере,
без iOS-specific рендеринга.

Решение пользователя (зафиксировано, не переоткрывается): **превью хостит backend** — backend хранит
сгенерированный сайт (новые таблицы `projects` + `site_files`) и отдаёт его по временному URL.
**Деплой (публикация в публичный интернет) отложен** — в этом проходе только генерация + хранение + превью.

Критический риск: backend отдаёт **пользовательский (Claude-сгенерированный) HTML/JS**, который исполняется
в браузере. Без изоляции это вектор XSS/utility-abuse относительно API-origin, утечки чужих проектов,
path-traversal по хранилищу.

## Decision

Backend отдаёт статику проекта по **временному signed URL** через выделенный публичный GET-эндпоинт,
строго изолированный от API-origin и от чужих проектов.

### 1. Хранение

- Контент файлов сайта хранится **в БД** (`site_files.content` `BYTEA`) на старте — простота, транзакционность,
  без внешней инфраструктуры ([03-data-model.md](../03-data-model.md)). Текстовые ассеты хранятся как UTF-8 байты,
  бинарные (изображения) — как есть.
- Лимиты (защита от раздувания БД, конфигурируемо через env):
  - размер одного файла ≤ **1 MB** (`PREVIEW_MAX_FILE_BYTES`);
  - суммарный размер проекта ≤ **10 MB** (`PREVIEW_MAX_PROJECT_BYTES`);
  - число файлов в проекте ≤ **200** (`PREVIEW_MAX_FILES`).
- Миграция на object-storage (S3-совместимое) — отложенный tech-debt [TD-009](../100-known-tech-debt.md), триггер —
  рост объёма/нагрузки. Интерфейс хранилища проектируется так, чтобы замена backend'а хранения не меняла tool/preview-контракты.

### 2. Signed URL (HMAC + TTL)

- Формат пути: `GET /v1/preview/{projectId}/{token}/{path:path}`.
- `token` — HMAC-SHA256 подпись под **отдельным** секретом `PREVIEW_URL_SECRET` (не пересекается с `JWT`/`KMS`/`ADMIN_API_SECRET`),
  покрывающая канонические поля: `projectId`, `ownerUserId`, `exp` (unix ts). Кодирование — url-safe base64 без паддинга;
  `exp` и подпись разделены так, чтобы проверялись вместе (например `<exp>.<hmac>`), верификация — constant-time (`hmac.compare_digest`).
- **TTL по умолчанию 15 минут** (`PREVIEW_URL_TTL_SECONDS`, конфигурируемо). Истёкший `exp` → `403`.
- Подделка подписи (несовпадение HMAC) → `403`. Подпись **не** покрывает `{path}` — один token открывает весь проект
  (его файлы), но **только этот** проект и **только** в пределах TTL; межпроектный доступ невозможен (см. §3).
- Токен выдаётся server-side tool'ом `site.preview` ([ADR-011](ADR-011-server-side-tools.md)) для проекта,
  принадлежащего пользователю сессии. Эндпоинт превью **сам по себе не требует пользовательского JWT** — авторизация
  заключена в валидном signed token (так превью открывается прямой ссылкой в браузере). Идентичность владельца
  «запечена» в подпись (`ownerUserId`), поэтому подделать доступ к чужому проекту нельзя.

### 3. Изоляция по владельцу/проекту

- `site_files` принадлежат `project`, `project` принадлежит `user` (FK-цепочка, [03-data-model.md](../03-data-model.md)).
- Эндпоинт превью читает файлы **только** для `projectId` из подписанного token и сверяет, что `projects.user_id`
  совпадает с `ownerUserId` из подписи. Несовпадение/отсутствие проекта → `403`/`404` (без раскрытия существования чужого проекта — `404`).
- Прочитать чужой проект нельзя: token подписан под конкретные `projectId`+`ownerUserId`, изменение любого из них ломает HMAC.

### 4. Защита отдаваемого контента (threat model)

Поскольку отдаётся пользовательский HTML/JS, эндпоинт превью применяет:

- **Path-traversal защита.** `{path}` нормализуется; запрещены `..`, абсолютные пути, NUL-байты, обратные слэши.
  Доступ только к `site_files` с точным совпадением нормализованного `path` в пределах проекта (lookup по `(project_id, path)`,
  не обращение к ФС). Несуществующий путь → `404`.
- **Content-type allowlist.** Отдаются только разрешённые типы: `text/html`, `text/css`, `text/javascript`,
  `application/json`, `image/png`, `image/jpeg`, `image/svg+xml` (SVG — с `Content-Security-Policy` и без скриптов —
  отдаётся как `image/svg+xml` с `Content-Disposition: inline` под общей CSP песочницы), `image/gif`, `image/webp`,
  `font/woff2`, `text/plain`. Content-type определяется по `site_files.content_type` (зафиксирован при записи tool'ом,
  валидируется по allowlist), не угадывается из расширения клиентом. Тип вне allowlist → файл не принимается на запись (`422`).
- **Заголовки безопасности (sandbox):**
  - `Content-Security-Policy: sandbox allow-scripts allow-forms; default-src 'self'; ...` — изоляция origin превью,
    запрет доступа к API-origin, ограничение источников. `sandbox`-директива лишает контент доступа к cookies/storage origin'а
    и к top-level navigation родителя.
  - `X-Content-Type-Options: nosniff` — запрет MIME-sniffing.
  - `X-Frame-Options: SAMEORIGIN` и `frame-ancestors 'self'` — контроль встраивания.
  - `Cache-Control: private, no-store` на превью-ответах.
  - **Никаких cookie/credentials**: эндпоинт не выставляет и не читает cookies, не участвует в пользовательской сессии.
- **Изоляция от API-origin (решение).** На старте превью отдаётся под **выделенным префиксом пути** `/v1/preview/*`
  с жёсткими sandbox-заголовками (выше) — самодостаточно для single-origin развёртывания. **Рекомендация для prod**
  (operational, не код): вынести превью на **отдельный поддомен/origin** (например `preview.<domain>`), чтобы даже при
  обходе CSP пользовательский JS не имел same-origin доступа к API. Это операционная рекомендация деплоя; зафиксирована
  как [Q-010-3](../99-open-questions.md) (не блокер — sandbox-заголовки безопасны на одном origin).
- **Размерные лимиты** — см. §1 (на запись).

## Consequences

**Положительные:**
- Превью работает в любом браузере, без iOS-рендеринга.
- Доступ авторизуется самодостаточным signed URL (прямая ссылка), при этом межпроектная/чужая изоляция гарантирована подписью.
- Threat model для отдачи пользовательского HTML/JS зафиксирована: sandbox CSP, nosniff, allowlist, path-traversal guard, TTL.
- Хранение в БД — без новой инфраструктуры, транзакционно с tool-операциями.

**Отрицательные / ограничения:**
- Контент в БД раздувает её при больших проектах — ограничено лимитами + [TD-009](../100-known-tech-debt.md) (миграция в object-storage).
- Single-origin sandbox слабее, чем отдельный поддомен; рекомендация поддомена — операционная ([Q-010-3](../99-open-questions.md)).
- Деплой/публикация в интернет вне scope — превью не является публичным хостингом.

## Alternatives

1. **iOS-хостинг превью (рендер на клиенте).** Отвергнуто решением пользователя: должно работать в любом браузере.
2. **Публичный (неподписанный) URL.** Отвергнуто: утечка чужих проектов, нет TTL, нет контроля доступа.
3. **Object-storage + CDN с первого дня.** Отвергнуто на старте: лишняя инфраструктура; БД-хранение достаточно
   при текущих лимитах. Зафиксировано как [TD-009](../100-known-tech-debt.md).
4. **JWT-авторизация превью (пользовательский Bearer).** Отвергнуто: превью открывается прямой ссылкой в браузере,
   браузер не шлёт Bearer на статику; signed URL — стандартное решение для temp-доступа к статике.
