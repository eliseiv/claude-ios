# ADR-007 — Ленивый провижининг users из доверенного JWT subject

- Статус: Accepted
- Дата: 2026-05-25
- Связан с: [ADR-001](ADR-001-stack-choice.md) (стек/монолит), [05-security.md](../05-security.md), [03-data-model.md](../03-data-model.md), [modules/api-gateway/03-architecture.md](../modules/api-gateway/03-architecture.md)

## Context

Идентичность пользователя приходит **исключительно** из внешнего JWT: claim `sub` = `userId` (UUID). Endpoint регистрации в системе нет и не предусмотрен — источник истины идентичности — доверенный JWT issuer (RS256, см. [05-security.md](../05-security.md), [Q-005-1](../99-open-questions.md)).

Все доменные таблицы имеют FK на `users(id)`: `subscriptions`, `wallets`, `byok_keys`, `ledger_transactions`, `chat_sessions`, `chat_steps` (через session), `tool_calls` (через session), `audit_logs` ([03-data-model.md](../03-data-model.md)). При этом **нигде не создаётся строка `users` при первом аутентифицированном доступе**.

Live e2e (BUG-1, CRITICAL) выявил: первая FK-зависимая вставка любого write-flow (`chat/run`, `subscription/sync`, `byok/set|toggle|delete`, `wallet/consume`) падает с `asyncpg.ForeignKeyViolationError` → HTTP 500. Весь write-path неработоспособен; 0 строк персистится. Unit-тесты сеяли `users` напрямую (фикстуры), поэтому дефект не был обнаружен на пирамиде.

Требуется механизм, который гарантирует существование строки `users` для верифицированного `sub` **до** любой FK-зависимой операции, единообразно для всех write-эндпоинтов, без race и без отдельного endpoint регистрации.

## Decision

**Ленивый провижининг (lazy provisioning).** Пользователь создаётся лениво — при первом аутентифицированном запросе на основе доверенного `sub` из верифицированного JWT. Отдельного registration-flow нет.

1. **Где.** Централизованно в API Gateway, в FastAPI-зависимости `get_current_user` — **после** успешной верификации JWT (подпись/`exp`/`iss`/`aud`) и извлечения `sub`, и **до** того, как `sub` используется в любом downstream use-case (chat/wallet/subscription/byok/policy). Это единственная точка, через которую проходят все аутентифицированные `/v1/*` запросы (middleware-шаг 3, [api-gateway/03-architecture.md](../modules/api-gateway/03-architecture.md)), поэтому провижининг покрывает все write-эндпоинты единообразно, без дублирования в каждом flow.

2. **Как (идемпотентный атомарный upsert).**
   ```sql
   INSERT INTO users (id) VALUES (:sub)
   ON CONFLICT (id) DO NOTHING;
   ```
   `ON CONFLICT (id) DO NOTHING` атомарен на уровне PostgreSQL — конкурентные запросы того же `sub` (несколько устройств/параллельные запросы) не вызывают race и не падают на дубликате PK. Прочие колонки `users` (`created_at`, `trial_used`) заполняются дефолтами DDL ([03-data-model.md](../03-data-model.md) §1), их значения провижинингом не трогаются.

3. **Транзакционность.** Upsert выполняется в рамках запроса до открытия доменной транзакции write-flow (или в её начале, в той же сессии БД), так что к моменту первой FK-зависимой вставки строка `users` гарантированно существует и видима. Так как `DO NOTHING` не меняет существующую строку, повторный провижининг при каждом запросе безопасен и дёшев (один INSERT, чаще всего no-op).

4. **Идентичность.** `users.id` ≡ JWT `sub`. UUID генерирует доверенный issuer (а не БД `gen_random_uuid()` на этом пути): на write-path значение `id` всегда задаётся явно из `sub`. Дефолт `gen_random_uuid()` в DDL сохраняется как технический fallback и для возможных служебных вставок, но доменный путь идентичности — issuer.

5. **Доверие к issuer.** Провижининг происходит **только** после полной верификации JWT. Невалидный/просроченный токен → `401` до провижининга — несуществующий/поддельный `sub` строку не создаёт. Сверка `userId` тела с `sub` (`403`) остаётся в силе ([05-security.md](../05-security.md)).

## Consequences

**Положительные:**
- Весь write-path работоспособен: FK на `users` всегда удовлетворён к моменту вставки.
- Нет endpoint регистрации — меньше API surface, идентичность централизованно делегирована trusted issuer.
- Единая точка (`get_current_user`) — нет риска забыть провижининг в новом write-flow.
- Race-free by construction (`ON CONFLICT DO NOTHING` атомарен).

**Отрицательные / ограничения:**
- На каждый аутентифицированный запрос — один дополнительный INSERT (в подавляющем большинстве no-op). Накладные расходы минимальны; при необходимости оптимизация (например, краткоживущий кэш «provisioned `sub`» в памяти процесса) — потенциальный tech-debt, **не требуется на старте**. Регистрировать TD не нужно: текущая стоимость приемлема.
- Строка `users` создаётся даже для read-only обращений (например, `GET /v1/policy/effective`, `GET /v1/wallet`). Это намеренно и безвредно: read-flow на пустом пользователе возвращают консистентные дефолты (нет подписки, баланс 0, trial не использован), а наличие строки делает поведение единообразным.

**Тестирование:**
- Unit/integration: ввести сценарий, где `users` **не** засеян предварительно — первый аутентифицированный запрос должен сам создать пользователя (регресс против BUG-1). Запрет на «фикстуру, всегда сеющую users» для провижининг-тестов.
- E2E: новый `sub`, никогда не обращавшийся, делает первый write-запрос (`chat/run`/`subscription/sync`) — должен пройти без 500 (см. E2E-PROV в [09-e2e-testing.md](../09-e2e-testing.md)).

## Alternatives

1. **Провижининг в начале каждого write-flow.** Отвергнуто: дублирование логики в 5+ местах, легко забыть в новом flow, выше риск рассинхрона. Централизация в `get_current_user` строго лучше.
2. **Отдельный endpoint регистрации (`POST /v1/users`).** Отвергнуто: добавляет шаг в клиентский flow, источник истины идентичности — issuer, регистрация дублировала бы `sub`; противоречит «identity из JWT».
3. **Создание users реактивно — ловить `ForeignKeyViolationError` и ретраить со вставкой users.** Отвергнуто: усложняет каждый flow, маскирует ошибки, хуже наблюдаемость; proactive upsert проще и предсказуемее.
4. **`ON CONFLICT (id) DO UPDATE`.** Отвергнуто: `DO NOTHING` достаточно (нет полей для обновления при провижининге), `DO UPDATE` создал бы лишний write и риск перетереть `trial_used`/`created_at`.
