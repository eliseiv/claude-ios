# ADR-046 — Структурное логирование исхода обработки Adapty-вебхука

- Статус: Accepted
- Дата: 2026-06-30
- Связано: расширяет [ADR-029](ADR-029-adapty-subscription-webhook.md) (Adapty subscription webhook); адресует **часть** [Q-029-3](../99-open-questions.md) (наблюдаемость, **не** маппинг событий); [05-security.md §Логирование](../05-security.md#логирование-безопасное), [05-security.md §Adapty webhook-авторизация](../05-security.md).
- Тип: implementation-ADR (только наблюдаемость; контракт, биллинг, схема данных, маппинг событий — без изменений).

## Контекст

Инцидент (прод `broadnova`): пользователь оформил подписку через промокод, **кредиты не начислены**. Adapty доставил вебхуки на `POST /v1/billing/adapty/webhook`, эндпоинт вернул `200`, но обработка прошла как `ignored` — строка в `adapty_webhook_events` не создана, грант не выполнен.

Первопричину **невозможно установить из логов**: `AdaptyWebhookService.handle()` возвращает причину (`reason`) исхода **только в теле HTTP-ответа** Adapty (`AdaptyWebhookResponse{result, reason, event_type}`) и **нигде её не логирует**. Тело ответа Adapty оператору недоступно. Это слепое пятно наблюдаемости: при `result="ignored"` нельзя отличить `user_not_found` от `missing_customer_user_id` от неизвестного `event_type` — то есть нельзя понять, было ли это потерянное начисление или безобидный проверочный пинг.

Исходы `handle()` (см. [billing-adapty/03-architecture.md](../modules/billing-adapty/03-architecture.md#матрица-ответов-точные-кодыстатусы)):
- `result ∈ {ignored, duplicate, applied}`;
- `reason ∈ {empty_body, invalid_json, not_an_object, missing_event_id, missing_customer_user_id, user_not_found, None}`;
- неизвестный `event_type` → `result="ignored"`, `reason=None`, `event_type` заполнен (эхо);
- `_apply()` → `applied` | `duplicate`.

Ключевое для инцидента: реальные строковые имена событий Adapty **ещё не сверены** с `KNOWN_EVENTS` ([Q-029-3](../99-open-questions.md) открыт). Если реальное имя события не входит в четыре дефолтных (`subscription_started/renewed/cancelled/expired`), валидное событие подписки **молча** уходит в `ignored` — корректно по контракту (fail-safe парсинг), но **без эффекта и без сигнала**. Именно этот класс отказа должен стать видимым.

## Решение

Логировать **каждый** исход `handle()` ровно одной структурной записью, по существующему паттерну проекта (`app.observability.logging.log_event`, образец `app.chat.orchestrator` «policy_decision»). HTTP-семантика, логика начисления, `KNOWN_EVENTS`, контракт `AdaptyWebhookResponse`, схема данных — **не меняются**.

### Где логировать

В **сервисе** (`src/app/billing_adapty/service.py`), не в роутере. Обоснование: только в сервисе доступны одновременно все `reason`, `event_id` и `customer_user_id`. `WebhookOutcome`, который видит роутер, **не несёт** `event_id`/`customer_user_id` (а на исходах `duplicate`/`applied` — и `event_type`), поэтому из роутера полную запись собрать нельзя. Роутер **не логирует** (иначе двойная запись).

Логгер: `logger = logging.getLogger(__name__)` в модуле сервиса (= `app.billing_adapty.service`). Эмиссия — через `log_event(logger, level, "adapty_webhook_outcome", **fields)`. Каждый вызов `handle()` порождает **ровно одну** запись.

### Allowlist полей лога (PII / секреты)

Логируется **только** фиксированный allowlist:

| Поле лога (camelCase) | Источник | Почему безопасно |
|---|---|---|
| `result` | `WebhookOutcome.result` | внутренний enum `ignored\|duplicate\|applied` |
| `reason` | `WebhookOutcome.reason` | внутренний enum причин (см. выше), не пользовательские данные |
| `eventType` | распарсенный `event_type` (lower) | идентификатор события Adapty, не PII |
| `eventId` | распарсенный `event_id` | непрозрачный идентификатор события Adapty, не PII |
| `customerUserId` | распарсенный `customer_user_id` | **наш внутренний `userId` (UUID)** — адресат гранта, не email/имя/телефон |

**Запрещено логировать** (явно): сырой payload целиком (`raw`/`body`), секрет авторизации Adapty (`Authorization`/bearer/`ADAPTY_WEBHOOK_SECRET`), любые поля payload вне allowlist (в т.ч. `vendor_product_id`, `expires_at`, `profile.*` — не нужны для диагностики исхода и исключены ради минимизации). Секрет авторизации физически в записи не присутствует (он в заголовке, не в теле; парсер его не читает) и дополнительно покрыт redaction (`authorization` ∈ `_DENY_SUBSTRINGS`). Allowlist зафиксирован в [05-security.md](../05-security.md#логирование-безопасное).

`None`-поля (`eventId`/`customerUserId`/`eventType` могут быть `None` на ранних `reason`) не приводят к сбою: `JsonFormatter` отбрасывает ключи со значением `None` из итогового JSON — отсутствующее поле = «не распарсено». `customer_user_id` — объект `uuid.UUID` → приводится к `str(...)` перед логированием (иначе `json.dumps` упадёт).

### Таблица уровней (`result`/`reason` → level)

| `result` | `reason` | `event_type` | Level | Обоснование |
|---|---|---|---|---|
| `applied` | `None` | known | **INFO** | штатный успех (начисление/обновление подписки выполнено) |
| `duplicate` | `None` | (любой) | **INFO** | идемпотентный повтор доставки Adapty — ожидаемо, не ошибка |
| `ignored` | `user_not_found` | known | **WARNING** | валидный UUID, но нет строки `users` → **потенциально потерянное начисление** |
| `ignored` | `missing_customer_user_id` | `None` | **WARNING** | нет/невалидный адресат гранта → **потенциально потерянное начисление** |
| `ignored` | `None` (неизвестный `event_type`) | set (эхо) | **WARNING** | событие распознано-но-не-маппится → **потенциально потерянное начисление**; прямой индикатор расхождения [Q-029-3](../99-open-questions.md) |
| `ignored` | `empty_body` | — | **DEBUG** | пустой POST = health-/connectivity-проба Adapty; чистый шум, событие невозможно по определению |
| `ignored` | `invalid_json` | — | **INFO** | тело есть, но не JSON; обычно проба, но при реальном трафике — возможный симптом дрейфа payload |
| `ignored` | `not_an_object` | — | **INFO** | JSON-не-объект; как `invalid_json` |
| `ignored` | `missing_event_id` | — | **INFO** | объект без `event_id`; проверочный пинг или дрейф формы payload |

Обоснование группировки:

- **WARNING** — три исхода, означающие «Adapty аутентифицировался и прислал нечто событие-подобное, но кредит пользователю доставить не удалось». Это **ровно класс инцидента** (промо-подписка без токенов). Они должны подниматься в дашборды/алерты. Сюда же отнесён неизвестный `event_type` — именно он маскирует незакрытый [Q-029-3](../99-open-questions.md): если реальное имя события Adapty не входит в `KNOWN_EVENTS`, валидное начисление теряется молча — WARNING делает это видимым.
- **INFO** — `applied` (успех) и `duplicate` (идемпотентный повтор — не ошибка): нормальная операционная активность, видна на дефолтном уровне.
- **DEBUG vs INFO для мусора/пинга.** `empty_body` (полностью пустое тело) — однозначно низкосигнальная health-/connectivity-проба → **DEBUG** (видна только при включённом debug). Остальные (`invalid_json`, `not_an_object`, `missing_event_id`) несут *какое-то* содержимое и потому имеют диагностическую ценность при появлении под реальным трафиком (возможный дрейф формата payload) → **INFO** (видны по умолчанию), но **не** WARNING: распарсиваемого события/адресата нет, терять нечего — поднимать тревогу не на что. Баланс шум/сигнал: на дефолтном INFO видно всё, кроме чистых пустых проб; включение DEBUG раскрывает и их.

Различение «неизвестный `event_type`» (`reason is None` в ветке `ignored`) однозначно: единственный `ignored`-исход с `reason=None` — это эхо неизвестного типа (все прочие `ignored` несут строковый `reason`). Реализация выбирает WARNING именно по `reason is None` внутри `ignored`.

### Сообщение и форма записи

Единое `message="adapty_webhook_outcome"` для всех исходов (как единый `"policy_decision"` в оркестраторе); операторы фильтруют по `level`/`result`/`reason`. Запись проходит ту же redaction-middleware (`JsonFormatter`→`redact`); имена allowlist-полей не совпадают ни с одной denylist-подстрокой (`key`/`token`/`secret`/`password`/`authorization`/`credential`) — значения не маскируются.

## Ограничения (явно)

- **HTTP-семантика не меняется:** `200` на всё, кроме реального сбоя (`5xx`) и провала авторизации (`401`/`500`). Логирование — побочный эффект, не влияет на код ответа.
- **Логика начисления не меняется:** дедуп, upsert подписки, грант, audit — как в [ADR-029](ADR-029-adapty-subscription-webhook.md). Лог не добавляет мутаций.
- **`KNOWN_EVENTS` не меняется.** Реальные имена событий Adapty по-прежнему неизвестны → **[Q-029-3](../99-open-questions.md) остаётся ОТКРЫТЫМ.** Этот ADR делает расхождение **видимым** (WARNING на неизвестном типе), но **не закрывает маппинг**: сопоставление реальных имён событий Adapty с четырьмя дефолтными — отдельная задача после получения имён из Adapty Dashboard (см. [Q-029-3](../99-open-questions.md), [07-deployment.md](../07-deployment.md) prod-checklist Adapty).
- **Без миграции.** Контракт `POST /v1/billing/adapty/webhook` (`AdaptyWebhookResponse`) не меняется. Провайдер-абстракция LLM не затронута. Биллинг неизменен.
- **Без новых env.** Уровень детализации управляется существующим глобальным log-level (`configure_logging`); отдельного флага не вводится.

## Последствия

Положительные:
- Каждый исход вебхука трассируем из логов: при повторении инцидента оператор сразу видит `result`/`reason`/`eventType`/`customerUserId` и причину «проглатывания».
- Неизвестный `event_type` (главный риск незакрытого [Q-029-3](../99-open-questions.md)) поднимается как WARNING → алертится до того, как пользователь пожалуется.
- Минимальная дельта: один helper + маршрутизация существующих `return`-точек через него; control-flow и контракт без изменений.

Отрицательные / риски:
- `empty_body` на DEBUG: при диагностике connectivity-проб нужно временно поднять log-level до DEBUG (приемлемо — это редкая операция).
- WARNING на `user_not_found`/`missing_customer_user_id` при шумном/ошибочном трафике Adapty может давать всплески предупреждений. Приемлемо: это именно тот сигнal, который нужен; фильтрация/алерт-пороги — на стороне log-стека.
- Лог-объём растёт на одну запись на вызов вебхука. Объём вебхуков подписок невелик — пренебрежимо.

## Альтернативы (отклонены)

1. **Логировать в роутере.** Отклонено: роутер видит только `WebhookOutcome` без `event_id`/`customer_user_id` (и без `event_type` на `duplicate`/`applied`) — полную диагностическую запись не собрать. Расширять `WebhookOutcome` ради лога — лишняя связанность ради того, что естественно живёт в сервисе.
2. **Логировать сырой payload целиком на `ignored`.** Отклонено: нарушает минимизацию PII/данных, риск утечки персональных данных и токенов из тела; allowlist полей даёт достаточную диагностику без сырого тела.
3. **Всё на INFO (без WARNING).** Отклонено: теряется сигнал — потерянное начисление неотличимо по уровню от штатного успеха; алертинг невозможен без парсинга `reason`.
4. **Новый env-флаг детализации логов вебхука.** Отклонено: избыточно; глобального log-level (INFO по умолчанию, DEBUG для пингов) достаточно. Без новых конфигов — меньше поверхность мис-конфигурации.
5. **Персистить причину в `adapty_webhook_events` (таблица/колонка).** Отклонено для этой итерации: требует миграции и записи даже на `ignored` (где сейчас мутаций нет — by design ADR-029). Логов достаточно для наблюдаемости; персист-аудит игнорируемых событий — потенциальная будущая задача, вне scope.
