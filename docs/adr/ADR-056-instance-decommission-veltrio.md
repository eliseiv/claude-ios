# ADR-056 — Вывод инстанса из эксплуатации на общем Traefik (`veltrio` / `veltriohub.shop`)

- **Статус:** Accepted (2026-07-18)
- **Расширяет:** [ADR-017 §Мульти-инстанс п.15](ADR-017-shared-server-traefik-deploy.md) (INSTANCES-loop). Тело ADR-017 не переписывается (immutability) — процедура вывода инстанса и её инвариант зафиксированы здесь.
- **Playbook:** [07-deployment.md §Мульти-инстанс](../07-deployment.md#мульти-инстанс--клонирование-сервиса), [§CI/CD INSTANCES-loop](../07-deployment.md#cicd-контракт-instances-loop-мульти-инстанс).

## Контекст

[ADR-017 §15](ADR-017-shared-server-traefik-deploy.md) ввёл `INSTANCES`-loop: deploy-job итерирует по списку `dir:project` и для каждого инстанса поднимает `api` с Traefik-labels, где Host-правило берётся из `SERVICE_DOMAIN` в `/opt/<dir>/.env`. Про **добавление** инстанса сказано явно; про **вывод** — одна строка «убрать из списка (каталог/тома чистятся вручную)», без разбора последствий.

2026-07-18 четвёртый инстанс `veltrio` (`veltriohub.shop`, OpenAI, [ADR-033](ADR-033-llm-provider-abstraction.md)) выведен из эксплуатации, а его домен **передан другому сервису на том же сервере** `87.239.135.154`, за тем же общим edge-Traefik (`/opt/edge`).

Фактическое состояние на момент решения:
- контейнеры `veltrio` остановлены и удалены (`docker compose -p veltrio -f docker-compose.prod.yml down`), `docker ps -a` по имени `veltrio` — пусто;
- Traefik-роутер снят: `https://veltriohub.shop/healthz` → `404` (прежде `200`); прежние labels — `traefik.http.routers.veltrio.rule=Host(veltriohub.shop)` на контейнере `veltrio-api-1`;
- тома `veltrio_pgdata` / `veltrio_redisdata` **сохранены**, каталог `/opt/veltrio` не удалён;
- бэкап снят: `/root/veltrio-decommission-20260718140433` (дамп БД — 20 таблиц, `.env`, `.secrets`);
- данные на момент отключения — тестовые: users=2, devices=2, subscriptions=0, ledger=1, chats=1;
- три остальных инстанса живы (`broadnova.shop` / `avelyraweb.shop` / `orvianix.shop` → `healthz` `200`).

**Опасность, которой не было при добавлении инстансов.** Пока домен принадлежал нам, повторный деплой выведенного инстанса был безвреден — он просто вернул бы сервис на свой домен. После передачи домена третьей стороне тот же деплой становится **перехватом чужого домена**: `up -d` пересоздаст контейнер с labels `Host(veltriohub.shop)`, Traefik подхватит роутер, и трафик нового владельца уйдёт к нам (плюс конкуренция за ACME-сертификат). Каталог `/opt/veltrio` и его `.env` с `SERVICE_DOMAIN=veltriohub.shop` сохранены — то есть **всё для такого перехвата на сервере уже лежит готовым**; единственное, что его предотвращает, — отсутствие записи `veltrio:veltrio` в `INSTANCES`.

## Решение

1. **Инстанс выводится удалением записи из `INSTANCES` в обоих workflow + `docker compose -p <proj> ... down` на сервере.** Обновление `INSTANCES` — в `.github/workflows/ci.yml` **и** `.github/workflows/deploy.yml` (ручной workflow идентичен gated deploy-job, иначе `workflow_dispatch` воскресит инстанс). `down` **без `-v`** — тома и данные сохраняются.

2. **Операционный инвариант (И-1, КРИТИЧНО): запись выведенного инстанса, чей домен передан другому владельцу, НЕ возвращается в `INSTANCES`.** Ни для «разового проверочного деплоя», ни для «поднять на 5 минут посмотреть данные». Причина — не косметика: деплой пересоздаёт Traefik Host-роутер по `SERVICE_DOMAIN` из сохранённого `.env` и отбирает домен у нового владельца.

3. **Возврат возможен только через смену домена.** Если инстанс когда-либо понадобится: сперва изменить `SERVICE_DOMAIN` (и `JWT_ISSUER`) в `/opt/veltrio/.env` на новый, свободный домен с A-записью → `87.239.135.154`, и только потом добавлять запись в `INSTANCES`. Возврат **прежнего** домена — только по согласованию с владельцем сервера и новым владельцем домена.

4. **Данные сохраняются, отключение обратимо.** Тома, каталог и бэкап не удаляются: инстанс восстановим (с новым доменом по п.3). Удаление томов/каталога/бэкапа — отдельное операторское решение, в этом ADR не принимается.

5. **Доступ к данным выведенного инстанса — без поднятия `api`.** Если понадобится прочитать БД, поднимать полный стек не нужно и нельзя (`api` тянет за собой Traefik-labels): достаточно бэкапа `/root/veltrio-decommission-20260718140433` либо запуска одного сервиса `postgres` (`docker compose -p veltrio -f docker-compose.prod.yml up -d postgres`) — у него нет ни labels, ни сети `web`.

6. **Наблюдаемость инварианта И-1 — по инварианту, а не по имени `veltrio`.** Инвариант общий (любой выведенный инстанс, чей домен передан другому владельцу), поэтому и наблюдатели формулируются через **реестры**, а не через литерал экземпляра. Реестры: нормативная таблица действующих инстансов и таблица **«Выведен из эксплуатации»** — обе в [07-deployment.md §CI/CD INSTANCES-loop](../07-deployment.md#cicd-контракт-instances-loop-мульти-инстанс). Автоматической проверки нет; оба наблюдателя выполняются вручную при подозрении и являются **разными** артефактами:
   - *наблюдатель срабатывания защиты* (**сравнение со списком действующих, а не отсутствие конкретной строки**): значение `INSTANCES` в `.github/workflows/ci.yml` и в `.github/workflows/deploy.yml` **посимвольно равно** нормативному значению из таблицы действующих инстансов. Любая лишняя запись — нарушение, независимо от того, какой инстанс она называет; новый выведенный инстанс покрывается этим наблюдателем автоматически, без правки проверки.
   - *наблюдатель отказа инварианта* (**авторитетный**; строится от следствия в данных, а не от имени контейнера): для **каждого** домена из таблицы «Выведен из эксплуатации» — `curl -sI https://<домен>/healthz` **не должен** отвечать нашим сервисом (наш ответ = `200` от `api`; ожидаемо — `404`/ответ нового владельца). Он инвариантен к именованию: ловит возврат под любым `-p <project>` и любым именем каталога (например `/opt/veltrio` под `-p veltrio-restore`), а также ручной `docker compose up` в обход CI.
   - *вспомогательный (не авторитетный)*: `docker ps --filter name=<project>-api` на сервере. Помогает локализовать нарушение, но **не** является доказательством его отсутствия — обходится переименованием project-имени.

7. **Скоуп.** Решение — операционное. Код приложения, контракты API, схема БД, миграции, `docker-compose.prod.yml` и остальные три инстанса **не затрагиваются**. Затронутые артефакты: `INSTANCES` в обоих workflow; [07-deployment.md](../07-deployment.md) (§Мульти-инстанс, §CI/CD INSTANCES-loop, prod-checklist); **`.env.prod.example`** — образец описывал `veltrio` как действующий инстанс и цель провижининга; правится devops. **Скоуп правки задан свойством, а не перечнем строк:** (а) ни один выведенный инстанс не назван действующим или целью провижининга; (б) перечни инстансов **поимённые** — сокращения (`…`, `и др.`, `each/every instance`) запрещены, поскольку элизия скрывает носителя от поиска по имени инстанса; (в) блок-предупреждение о выведенном инстансе (DECOMMISSIONED) **разрешён** — он запрещает провижининг, а не перечисляет действующих. Строки `58`, `75`, `235`, `267` — **иллюстрация на момент решения, перечень не исчерпывающий**: правка затронула и носитель, где выведенный инстанс подразумевался через элизию, по имени не называясь. Требование зафиксировано в [07-deployment.md §Мульти-инстанс](../07-deployment.md#мульти-инстанс--клонирование-сервиса); реестр долга — `TD-030`. О сервисе, которому передан домен, нам известно только, что это другой сервис на том же сервере, — никаких требований к нему здесь не формулируется.

## Последствия

**Плюсы.** Мульти-инстанс-топология сокращается до трёх действующих инстансов, ресурсы сервера освобождены; данные и секреты выведенного инстанса не потеряны (бэкап + тома); опасность перехвата чужого домена названа явно и локализована в одном месте (`INSTANCES`).

**Минусы / риски.** Единственная защита от перехвата — отсутствие строки в `INSTANCES`, а всё остальное для деплоя на сервере сохранено; защита декларативная и держится на дисциплине (см. п.6 — автоматической проверки нет). Каталог `/opt/veltrio` с `SERVICE_DOMAIN=veltriohub.shop` остаётся на диске неограниченно долго, и его опасность со временем забывается.

> **Остаточный риск заведён в реестр: [TD-030](../100-known-tech-debt.md).** Отложенное решение из п.4 (удаление томов/каталога/бэкапа — за оператором) носителем имеет не текст этого ADR, а запись `TD-030` с критерием закрытия. Без неё риск жил бы только в документе, который никто не перечитывает.

Предупреждение продублировано в комментариях обоих workflow, в [§Мульти-инстанс](../07-deployment.md#мульти-инстанс--клонирование-сервиса), в [§CI/CD INSTANCES-loop](../07-deployment.md#cicd-контракт-instances-loop-мульти-инстанс) и в prod-checklist.

## Альтернативы

- **Удалить `/opt/veltrio`, тома и бэкап целиком.** Отклонено: устраняет риск перехвата полностью, но безвозвратно уничтожает данные инстанса ради экономии дискового места; отключение решили сделать обратимым. При решении удалить — требуется отдельное операторское подтверждение, риск перехвата отпадает вместе с каталогом.
- **Оставить запись в `INSTANCES`, но обнулить `SERVICE_DOMAIN` в `/opt/veltrio/.env`.** Отклонено: деплой продолжал бы собирать образ и поднимать контейнеры выведенного инстанса (расход ресурсов, шум в `$FAILED`), а защита держалась бы на содержимом `.env` на сервере, которое не видно в репозитории и не ревьюится.
- **Сменить `SERVICE_DOMAIN` на заглушку и оставить инстанс живым.** Отклонено: инстанс не нужен, а живой стек требует свободного домена, ACME-сертификата и попадает под prod-checklist на общих основаниях.
- **Не заводить ADR, ограничиться правкой `07-deployment.md`.** Отклонено: вводится новый операционный инвариант класса «вывод инстанса на общем Traefik», применимый к любому будущему инстансу, и он ревизует однострочную формулировку [ADR-017 §15](ADR-017-shared-server-traefik-deploy.md) про удаление из списка. Решения такого рода живут в реестре ADR, иначе следующий оператор их не найдёт.
