# Profile — Architecture

## Размещение
Пакет `src/app/profile/`: репозиторий над `users` + use-cases (get/update) + роутер `/v1/profile`.

## Вычисление accountId (BR-PR-1)
- Детерминированная функция `account_id(user_id: UUID) -> str`, формат `XXXX-XXXX-XXXXX` (две группы по 4 цифры + группа из 5 буквенно-цифровых, как `8472-1936-AXQ5`).
- Алгоритм: взять стабильный хэш `user_id` (например первые байты SHA-256 от канонической строки UUID), отобразить в фиксированный алфавит/длины групп. **Чистая функция, без БД-записи** — один и тот же `user_id` всегда даёт один `accountId`.
- Это **отображение для человека**, не секрет и не вторичный ключ авторизации: авторизация всегда по JWT `sub`. Коллизии теоретически возможны, но `accountId` нигде не используется как идентификатор для доступа (только отображение) — поэтому уникальность не критична. Если в будущем потребуется гарантированно уникальный публичный id — отдельное решение (не в этом проходе).

## Запись displayName
- `PATCH` → `UPDATE users SET display_name = :name WHERE id = :sub`. `users` строка существует (lazy provisioning, [ADR-007](../../adr/ADR-007-lazy-user-provisioning.md)).

## Инварианты
- `accountId` не хранится (нет колонки) — устраняет рассинхрон БД↔вычисление.
- Profile не выдаёт чужие данные: всё скоупится `sub`.
