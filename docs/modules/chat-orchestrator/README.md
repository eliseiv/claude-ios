## DoD
Policy Engine вызывается перед каждой генерацией; tool-loop стабилен на нескольких шагах; tool payload строго типизирован; status=blocked машиночитаем; usage учитывается; mode=byok использует ключ пользователя; mode=credits инициирует списание.

- 2026-06-04: BUG-5 (CRITICAL) — реконструкция диалога server-side tool-loop (`site.*`) ломалась с реальным Claude. `tool_use` и `tool_result` пишутся в одной транзакции → равный transaction-time `created_at`; `list_steps` сортировал по `(created_at, id)`, UUID-tie-break случайно ставил `tool_result` раньше `tool_use` → orphan tool_result → Anthropic `400` → `502`. Client-side не затронут (разные транзакции). Решение ([ADR-021](../../adr/ADR-021-deterministic-step-order-and-block-normalization.md)): новая колонка `chat_steps.seq BIGINT GENERATED ALWAYS AS IDENTITY` (монотонный порядок вставки); `list_steps`/`next_step_after` сортируют по `seq`, НЕ `created_at`; индекс `ix_steps_session_seq`. Дополнительно — нормализация content-блоков перед персистом: служебное SDK-поле `caller` (из `block.model_dump()`) вырезается, в `chat_steps.payload`/реплее только wire-валидные поля Anthropic (raw `tool_use.id` сохраняется, ADR-008 цел). Миграция `0006` (add column + backfill по `(created_at,id)` + NOT NULL + индекс), цепочка `0001`→`0006`. Контракт `/chat/run` наружу НЕ меняется. См. [03-architecture.md §Детерминированный порядок шагов](03-architecture.md#детерминированный-порядок-шагов-и-нормализация-payload-adr-021), [04-data-model.md](04-data-model.md), [09-testing.md](09-testing.md). Scope backend + qa.
