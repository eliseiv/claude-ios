### Q-024-1 — сопутствующий текст при `tool_call` (run↔история)
Нестыковка 3 из [ADR-024](adr/ADR-024-history-payload-domain-normalization.md): assistant-ход может нести в одном сообщении и `text`, и `tool_use`; история отдаёт оба блока шага (после доменной нормализации), а `ChatResponse` при `status=tool_call` возвращает только `toolCall`, теряя текст. ADR-024 **на стороне истории** уже даёт полный шаг (клиент берёт текст из `steps[].payload` по `stepId`). Остаётся продуктовый вопрос — дотягивать ли сам `ChatResponse`:
- **Вариант A (рекомендован):** добавить опц. `assistantMessage` в ответ при `status=tool_call`; backend перестаёт отбрасывать текст (`orchestrator.py:661`), кладёт его в `assistantMessage`. Плюс: прогресс-UI сразу показывает «Claude сказал X и вызвал инструмент Y» без запроса истории. Аддитивно (поле уже опционально).
- **Вариант B:** не менять `ChatResponse`; задокументировать, что полный ход доступен в истории по `stepId`/`messageStepId` ([ADR-023](adr/ADR-023-sync-ids-in-chat-response.md)). Минус: доп. round-trip/задержка для показа сопутствующего текста.

**Resolution (2026-06-10, решение пользователя = вариант A).** При `status=tool_call` `ChatResponse` (`/chat/run` и `/chat/tool-result`) **опционально** возвращает `assistantMessage` — текст из `text`-блоков **того же** assistant-шага, чей `tool_use` вернулся как `toolCall` (тот же шаг, на который указывает `stepId`, [ADR-023](adr/ADR-023-sync-ids-in-chat-response.md)). Значение = конкатенация/текст этих `text`-блоков; при их отсутствии `assistantMessage = null`/опущено. Backend перестаёт отбрасывать текст (`orchestrator.py:661`) и кладёт его в `assistantMessage`. Согласовано с нормализацией истории: текст в `GET /v1/chats/{id}` → `steps[].payload.content[]` того же шага (по `stepId`) — **тот же**. Изменение аддитивно/обратносовместимо: поле уже было опционально-nullable; меняется лишь то, что оно теперь может быть НЕ-null при `tool_call`; остальные поля `tool_call` (`toolCall` обязателен, `toolCall.id`/`name` доменные) не меняются. `assistantMessage` при `status=assistant_message` (финал) — без изменений; при `status=blocked` — `null`. Cross-ref: [ADR-024 §Decision п.3](adr/ADR-024-history-payload-domain-normalization.md), [chat-orchestrator/02-api-contracts.md §Response (200)](modules/chat-orchestrator/02-api-contracts.md#response-200), [chat-orchestrator/09-testing.md §ADR-024](modules/chat-orchestrator/09-testing.md), [API-REFERENCE.md §5](API-REFERENCE.md).

### Инцидент RU-флоу «оплата есть — начисления нет» — Resolved ([ADR-053](adr/ADR-053-cloudpayments-webhook-user-resolution-via-auth-devices.md), 2026-07-03)
**Корень:** CloudPayments-вебхук резолвил пользователя из `AccountId`/`Data.user_id` **только в `users`**, но broadapps на RU-флоу шлёт **deviceId** (id устройства, напр. `55cbe083-...`), которого в `users` нет → `user_not_found` → оплата без начисления. Связь deviceId→userId (`55cbe083-...`→`b0f407bd-...`) есть в **нашей** `auth_devices` ([ADR-018](adr/ADR-018-embedded-auth-issuer.md)). **Фикс ([ADR-053](adr/ADR-053-cloudpayments-webhook-user-resolution-via-auth-devices.md)):** двухступенчатый резолв в `service.py` — (a) `X∈users`→`userId=X`; (b) иначе `X∈auth_devices.device_id`→связанный `user_id`; (c) иначе `user_not_found`. Начисление — на резолвнутый `userId`; маппинг только из нашей БД; наблюдаемость `resolvedVia`. Скоуп только CloudPayments-вебхук.
**Операторская заметка (iOS, НЕ наш код):** sync-механизм RU-оплаты = серверный колбэк broadapps→наш вебхук; клиентский вызов `/checkout` ([ADR-051](adr/ADR-051-cloudpayments-checkout-payment-link.md)) как «синхронизация» избыточен — создаёт **дубль-платёж**, а не подтверждает уже прошедшую оплату. Поведение checkout не меняется. Остаточная сверка регистра deviceId — [Q-053-1](#открытые-вопросы-cloudpaymentsbroadapps-webhook-ru-путь-2026-07-02-adr-050).
